# 要件定義書

## 0. 文書情報
- 文書名: 日本賃貸エージェント 要件定義書
- 版数: v0.1 (PoC)
- 作成日: 2026-04-06
- ベース技術方針: `AI-Scientist-v2` のオーケストレーション層（Stage管理 / Journal管理 / Tool抽象）を流用し、研究実験特化部分を業務フロー向けに置換する。

---

## 1. 背景と課題
日本の賃貸不動産の意思決定は、以下の業務が分断されているため、ユーザー負荷が高い。

- 物件探索
- 重複掲載の見分け
- 初期費用比較
- 条件確認
- 不動産会社とのやり取り
- 内見予約
- 契約前チェック

ユーザーの本質的な目的は「自分に合う部屋を、損せず、迷わず、早く契約したい」であるが、現状のUXはその目的に対して非効率である。

---

## 2. 目的
### 2.1 目的
「検索 -> 比較 -> 行動（問い合わせ） -> 契約前リスク確認」を単一エージェント体験として統合する。

### 2.2 PoCでの達成目標
- 重複物件統合が可能であること
- ユーザー条件に基づく推薦と理由説明が可能であること
- 実用的な問い合わせ文を自動生成できること
- 契約前の注意点抽出が可能であること

---

## 3. スコープ
### 3.1 対象（In Scope）
- 日本賃貸を想定した物件データの正規化・重複統合
- 条件ベースのランキングと説明
- 不動産会社向け問い合わせ文生成（日本語）
- 契約文・費用表からの注意点抽出
- ユーザー/タスクメモリ管理

### 3.2 非対象（Out of Scope）
- 実際の賃貸媒体への本番スクレイピング運用
- 契約締結の法的自動化
- 重要事項説明の法的代替
- 本番決済/電子署名連携

---

## 4. システム全体像
### 4.1 テーマ
日本の賃貸不動産における、分断された情報・手続き・意思決定負荷を統合するエージェント。

### 4.2 5層エージェント構成
1. Planner
2. Search / Normalize Agent
3. Reasoning / Ranking Agent
4. Communication Agent
5. Contract / Risk Check Agent

### 4.3 本プロジェクトの実行パイプライン（PoC）
研究実験パイプライン（実験/論文中心）ではなく、賃貸意思決定支援パイプラインとして以下を採用する。

```
[1] 要件整理 (Planner)
          ↓ plan.json / user_memory.json / missing_slots
[2] 情報統合 (Search / Normalize)
          ↓ normalized_properties / duplicate_groups
[3] 推薦・比較 (Reasoning / Ranking)
          ↓ ranked_properties / why_selected / why_not_selected
[4] 行動支援 (Communication)
          ↓ message_draft / check_items / task_memory
[5] 契約前リスク確認 (Contract / Risk Check)
          ↓ risk_items / must_confirm_list
[6] 監査ログ・評価改善 (Audit / Evaluation Loop)
```

各ステップの判断根拠・中間生成物は監査可能な形で保存し、再実行時に追跡可能とする。

## 5. 基盤方針（AI-Scientist-v2 ベース）
### 5.1 流用するコンポーネント
- Stage遷移管理: `AgentManager` 相当
- 実行履歴/状態木: `Journal` / `Node` 相当
- ツール抽象: `BaseTool` 相当
- LLM関数呼び出し: `FunctionSpec` + backend層
- `AI-Scientist-v2` ディレクトリは参照・参考専用とし、フォルダー内ファイルは変更しない。
- 本アプリケーションの実装コードは本ディレクトリ配下に新規作成し、`AI-Scientist-v2` とは分離して管理する。
- 設計・実装判断時は `AI-Scientist-v2` の関連コードを繰り返し詳細に読み込み、構造と意図を十分に理解した上で反映することを推奨する。

### 5.2 置換するコンポーネント
- 研究実験コード生成・訓練・plot関連
- ML研究向けプロンプト（GPU/学習曲線前提）
- 論文執筆/査読フロー

### 5.3 新規実装するコンポーネント
- 物件正規化・重複統合ロジック
- 賃貸向けスコアリング/説明生成
- 問い合わせテンプレート最適化
- 契約リスク抽出器（ルール + LLM）

### 5.4 アプリケーション実装方針
- フロントエンドは `TypeScript` + `React` で実装する。
- バックエンドは `Python` + `FastAPI` で実装する。
- 利用するLLMのAPIモデルプロバイダーは `OpenAI`、`Gemini`、`Groq`、`Claude` とする。
- Web検索は `Brave Search API` を利用する。
- UIアーキテクチャは `Generative UI` を採用し、自然言語チャットを主要なユーザー導線とする。
- 比較表・リスク一覧・確認項目は都度生成する構造化UI（カード/テーブル/チェックリスト）で提示する。
- 問い合わせ送信などの重要操作は、実行前に明示確認ステップを必須とする。
- UIスタイリングは `Tailwind CSS` を標準採用する。
- 個別CSSは最小限に留め、デザインの一貫性は `Tailwind CSS` のユーティリティと設定で管理する。
- 開発環境は `docker compose` で一発起動できる構成とする（例: `docker compose up`）。
- 開発効率のため、フロントエンド/バックエンドの両方でホットリロードを有効化する。
- `API` キーなどの機密情報は `.env` で管理し、リポジトリには `.env.example` のみをコミットする。

### 5.5 UIレイアウト方針（チャット中心）
- ヘッダーは設けず、画面は `メッセージ表示領域` と `下部固定Composer` を基本構成とする。
- Composerは画面最下部に固定し、`テキストエリア` と `送信ボタン` を配置する。
- メッセージ表示領域はComposerと重ならないように下部余白を確保する。
- チャット本文とGenerative UIは中央寄せの単一カラムで表示し、横幅は広めの `max-width` で統一する。
- 比較表・リスク一覧・確認項目はチャット内に構造化UI（カード/テーブル/チェックリスト）として表示する。
- 問い合わせ送信などの重要操作は確認ステップを経たユーザー明示承認後に実行する。
- 本文上部に軽量ステータス表示（検索中/比較中/送信準備完了等）を置き、モバイルではSafe Area対応・入力欄自動伸長を必須とする。

### 5.6 AI-Scientist-v2 活用境界
- `AI-Scientist-v2` は参照専用とし、設計思想を基に本プロジェクトで新規実装する。
- 流用対象は `Stage` 管理、`Journal/Node` による状態管理、`FunctionSpec` ベースのツール呼び出し設計、実行ログ構造とする。
- 非流用対象は、研究実験コード自動生成/実行、学習曲線前提の評価、plot集計、論文執筆、論文査読とする。
- 本プロダクトの本番経路では、LLM生成コードの動的実行は行わない。

### 5.7 LLMプロバイダー実装方針（OpenAI/Gemini/Groq/Claude）
- LLM呼び出しは Provider Adapter 層を新規実装し、`OpenAI`、`Gemini`、`Groq`、`Claude` を同一I/Fで切り替え可能にする。
- 構造化出力（JSON Schema）を標準とし、Function Calling 非対応モデルでは JSON モード＋バリデーションで代替する。
- プロバイダー選択、モデル名、タイムアウト、リトライは `.env` で切り替える。

### 5.8 検索実装方針
- Web検索は `Brave Search API` を唯一の標準検索手段として利用する。
- ブラウザ自動操作は標準経路では使用せず、例外時の補助手段としてのみ扱う。

## 6. 機能要件

### FR-001 Planner（対話要件整理）
- ユーザーの曖昧要望から必須条件/希望条件を抽出する。
- 不足情報を質問し、次アクションを決定する。
- 出力: `plan`, `next_action`, `missing_slots`。

### FR-002 Search/Normalize（情報統合）
- 複数入力の物件情報を共通スキーマへ正規化する。
- 表記ゆれを補正し、同一物件候補をクラスタリングする。
- 出力: `normalized_properties`, `duplicate_groups`。

### FR-003 Ranking（推薦と説明）
- ユーザー条件に基づき候補を順位付けする。
- 各候補について「選定理由」「除外理由」を説明する。
- 出力: `ranked_properties`, `why_selected`, `why_not_selected`。

### FR-004 Communication（問い合わせ支援）
- 物件ごとの確認事項に応じた日本語問い合わせ文を生成する。
- 丁寧・簡潔・失礼のない文体を保つ。
- 出力: `message_draft`, `check_items`。

### FR-005 Contract/Risk Check（契約前確認）
- 初期費用表・契約文から注意点を抽出する。
- 更新料、短期解約違約金、解約予告、保証会社条件を検知する。
- 出力: `risk_items`, `must_confirm_list`。

### FR-006 User Memory
- 家賃上限、勤務地、駅距離、必須条件、妥協条件、嗜好を保持する。
- セッションを跨いだ再利用を可能にする。

### FR-007 Task Memory
- 問い合わせ済み、回答待ち、比較中、落選理由、契約候補を保持する。
- 意思決定の進行状態を可視化可能にする。

### FR-008 監査ログ
- 各提案の根拠（入力、スコア、判断理由）を保存する。
- 再現可能な意思決定トレースを提供する。

## 7. データ要件
### 7.1 物件共通スキーマ（PoC最小）
- `property_id_norm`
- `source_id`
- `building_name_norm`
- `address_norm`
- `layout`
- `area_m2`
- `rent`
- `management_fee`
- `deposit`
- `key_money`
- `station_walk_min`
- `available_date`
- `agency_name`
- `notes`

### 7.2 重複統合キー（候補）
- 建物名正規化 + 住所正規化 + 間取り + 面積近傍

### 7.3 入力
- 自然言語会話
- 物件情報テキスト
- 初期費用表
- 契約書PDF/テキスト
- 問い合わせ返信文

### 7.4 出力
- 条件整理
- 比較表
- 推薦理由
- 問い合わせ文
- 契約前注意点一覧

---

## 8. 非機能要件
### NFR-001 応答性能
- PoC目標: 単一ターン応答 10秒以内（外部I/O除く）

### NFR-002 説明可能性
- 推薦結果は必ず理由文字列を持つこと

### NFR-003 日本語品質
- 問い合わせ文は敬語として破綻しないこと

### NFR-004 可観測性
- 各エージェントの入力・出力・失敗理由をログ化すること

### NFR-005 安全性
- 契約判断は「最終判断はユーザー/専門家確認」の免責表示を行うこと

### NFR-006 UIデザイン品質
- UIは近代的・未来的で、美しいビジュアルデザインを備えること

---

## 9. 評価要件（PoC）
- 重複物件統合の精度（Precision/Recall/F1）
- 推薦の納得度（ユーザー評価）
- 問い合わせ文の実用性（人手評価）
- 契約リスク抽出の正確さ（正解ラベル比較）
- 意思決定時間の削減率（ベースライン比較）

---

## 10. 受け入れ基準
以下を満たした場合、PoC合格とする。

1. 重複統合データセットで重複検知F1が目標値を超える（目標値は実験開始時に固定）。
2. 同一ユーザー条件で再実行時、推薦上位結果と理由が再現可能である。
3. 問い合わせ文が実務上そのまま送信可能レベルと評価される。
4. 契約リスク抽出で重大項目（更新料/違約金/解約予告/保証会社）を漏らさない。
5. 検索 -> 比較 -> 問い合わせ -> リスク確認の一連フローが通る。
6. `AI-Scientist-v2` ディレクトリに差分がないこと（参照専用運用が守られていること）。
7. `OpenAI`、`Gemini`、`Groq`、`Claude` の4プロバイダーで基本対話と構造化出力が動作すること。
8. 検索機能が `Brave Search API` 経由で動作し、検索結果の正規化まで一連で通ること。
9. 本番経路で LLM 生成コードを実行しないこと（設定および実装で担保されていること）。

## 11. 実装フェーズ（PoC）
### Phase 1: 基盤移植
- `AI-Scientist-v2` の Stage/Journal/Tool 構造を参照し、本ディレクトリ上の新規コードとして賃貸ドメイン向けに再実装する。

### Phase 2: データ統合
- 架空賃貸データ作成、正規化、重複統合の実装。

### Phase 3: 推薦/説明
- 条件スコアリングと理由生成。

### Phase 4: 行動支援
- 問い合わせ文生成と確認項目生成。

### Phase 5: 契約チェック
- 契約条項抽出とリスク指摘。

### Phase 6: 評価
- 指標計測と改善サイクル。

---

## 12. リスクと対策
- データ品質不足
  - 対策: 正規化ルールと例外辞書を導入
- 同一物件判定誤り
  - 対策: ルール + 類似度のハイブリッド判定
- LLM出力ぶれ
  - 対策: 関数呼び出しJSON制約、テンプレート化、再試行
- 法的誤案内
  - 対策: 断定回避、確認推奨、監査ログ保持
