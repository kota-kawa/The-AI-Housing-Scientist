# Part 5 提出ドラフト

## タイトル
The AI Housing Scientist
AI Scientist 型の Tree Search と自動評価を、日本の賃貸物件調査エージェントへ拡張した実装

## 1. 概要
Part 5 では、Sakana AI の The AI Scientist にある中核的な発想を、機械学習論文生成ではない実務的ドメインへ移植することを目標にして、**The AI Housing Scientist** という PoC を構築した。

このシステムは、日本の賃貸物件探しを対象に、以下の流れを end-to-end で支援する。

1. ユーザーの希望条件を自然文から整理する
2. 不足条件を追加で質問する
3. 調査計画を提示し、ユーザー承認を得る
4. 複数の探索方針を持つ分岐型リサーチを実行する
5. 検索結果を正規化・重複排除・整合性審査する
6. 候補物件をランキング・比較する
7. 問い合わせ文面を下書きする
8. 契約条項テキストからリスクを抽出する

重要なのは、このプロジェクトが AI Scientist の既存コードをそのまま動かしたものではない点である。論文生成エージェントをそのまま使うのではなく、**planning, branching, evaluation, synthesis** という構造を、住宅探索という別ドメイン向けに作り直した。

## 2. Part 5 の趣旨との対応
私は Part 5 を、「Sakana AI の agentic research の考え方を理解したうえで、それを別の問題設定に意味のある形で拡張する課題」だと解釈した。その観点で、この実装は次の 3 点で課題趣旨に沿っている。

第一に、対象ドメインを明確に変えている。ここで扱っているのは ML 論文生成ではなく、日本の賃貸物件探索という、情報が不完全で、条件交渉が多く、実務的な意思決定が必要な領域である。

第二に、AI Scientist v2 の tree search 的な考え方を実装レベルで取り入れている。単発の検索や単発の回答ではなく、複数の探索分岐を走らせ、その分岐ごとに結果を評価し、弱い枝を刈り込み、最終的に最良の経路を採用する構造になっている。

第三に、出力が単なる「候補一覧」では終わらない。候補比較、推薦理由、問い合わせ文、契約リスク確認までつなげることで、探索から実行までを一つの agent loop としてまとめている。

つまりこのプロジェクトは、AI Scientist の「論文を書く」という表面的な用途ではなく、**探索を計画し、分岐を評価し、根拠をまとめて意思決定につなげる**という根本構造を、別領域で再実装したものだと位置づけている。

## 3. なぜ賃貸探索を題材にしたか
日本の賃貸物件探索は、agentic system の題材として相性が良い。理由は次の通りである。

- 情報源が検索結果、物件詳細、契約文面などに分散している
- データが古い、欠損している、矛盾していることが多い
- 厳しめの条件で深掘りする枝と、候補数を確保するために緩める枝の両方が必要になる
- 正解が一意ではなく、家賃、駅距離、間取り、安心感などのトレードオフになる
- 最終的には問い合わせや契約判断など、行動につながる出力が必要になる

この性質は、単発の LLM 応答よりも、複数の探索方針を試して評価する tree search 型の設計と相性が良い。したがって、AI Scientist 系の発想を「論文執筆」以外へ広げる題材として適していると考えた。

## 4. 実装したシステムの概要
本リポジトリは、フロントエンドとバックエンドを含むフルスタック構成の PoC である。

- Frontend: React 19, TypeScript, Vite
- Backend: FastAPI, Python 3.11
- State / Audit: SQLite
- Search: Brave Search API + mock property catalog
- LLM routing: planner, research, communication, risk_check の用途別ルーティング

ユーザー体験としては、以下の流れで動作する。

1. ユーザーが自然文で条件を入力する
2. planner が希望条件を構造化する
3. 足りない条件があれば structured question として聞き返す
4. 調査計画を提示し、ユーザー承認を待つ
5. 承認後、バックグラウンドで research job を開始する
6. 分岐探索により複数の検索戦略を試す
7. 候補を正規化・評価して、最終レポートと比較 UI を返す
8. 選択した候補に対して問い合わせ文を生成する
9. 契約条項を貼り付けると、更新料・短期解約違約金・解約予告・保証会社などのリスクを抽出する

## 5. 技術的な中核貢献
このプロジェクトの主な技術的貢献は、**AI Scientist 型の branching + evaluator loop を、住宅探索ドメインに移植したこと**である。

### 5.1 Planning を Research の前段に置いたこと
AI Scientist 的な探索を成立させるには、最初の問題定義が雑だと全体が崩れる。そのため、このシステムではまず planner が自然文を structured slots に落とし込む。

扱う主な slot は以下である。

- 物件種別
- 希望エリア
- 予算上限
- 間取り
- 駅徒歩
- 入居時期
- 必須条件
- あると良い条件

不足情報がある場合は follow-up question を返し、ある程度条件が固まると research plan を生成する。さらに、その plan は即実行せず、ユーザー承認を待つ設計にしている。

これは単なる UX 上の都合ではなく、探索の root を人間と agent が共同で確定するという意味で、後段の tree search の質を左右する重要な設計である。

### 5.2 Single-pass ではなく Branching Search にしたこと
このシステムの research は、単純な「検索してランキング」ではない。`HousingResearchAgentManager` が中心となり、複数の探索分岐を管理する。

state machine 上のトップレベル stage は以下の 3 つである。

- `plan_finalize`
- `tree_search`
- `synthesize`

その内部で、探索分岐ごとに異なる operator を適用する。実装されている代表的な operator は以下である。

- `tighten_match`: 条件一致度を優先して絞る
- `relax_for_coverage`: 候補数確保のために探索を広げる
- `source_diversify`: 情報源の偏りを下げる
- `detail_first`: 詳細ページの取得を優先する
- `schema_first`: 比較に必要な項目の充足を優先する
- `exploit_best`: 有望な枝をさらに深掘りする
- `explore_adjacent`: 近接条件・周辺条件へ広げる

この構造により、「条件一致は強いが候補数が少ない枝」「候補数は多いが詳細情報が薄い枝」などを別々に評価できる。これは AI Scientist v2 の tree search の発想を、住宅探索用に具体化したものだと考えている。

### 5.3 分岐実行を Tool 化したこと
各 branch は、内部で複数の tool を順番に実行する。`agent_manager_tooling_mixin.py` では、少なくとも次の処理が tool として切り出されている。

- `retrieve`
- `enrich`
- `normalize_dedupe`
- `integrity_review`
- `rank`

各 tool は schema validation を持ち、入力と出力の形が固定されている。この構造には 2 つの利点がある。

1. 分岐単位で何を実行したかを追跡しやすい
2. 一つの巨大プロンプトに依存せず、agent loop を保守しやすい

これは、AI Scientist 的な「モデルがすべてを一度に解く」のではなく、複数の処理を orchestrate するという発想に近い。

### 5.4 Reviewer 的な Branch Evaluation を入れたこと
このプロジェクトで最も AI Scientist らしい部分は、分岐をただ生成するだけでなく、**分岐を評価して選ぶ仕組み**を入れた点である。

`offline_eval.py` では、分岐ごとに以下のような指標を計算している。

- 正規化後の候補数
- ランキング可能候補数
- 詳細ページ補完率
- 構造化率
- 上位候補スコア
- source diversity
- duplicate ratio
- integrity drop ratio

これらを branch score にまとめ、弱い枝は prune し、改善幅のない枝も落とす。最終的に best branch を選ぶときも、「最後に生成された枝」ではなく、「評価が最も高い枝」を採用する。

つまりこのシステムでは、生成よりも**評価が制御権を持つ**。この点が、AI Scientist の reviewer / evaluator 的な考え方と強く対応している。

### 5.5 Evidence Reliability を重視したこと
賃貸領域では、検索ヒットがそのまま推薦候補になるとは限らない。古い掲載、価格矛盾、賃貸と売買の混在、希望エリア外の物件などが普通に混ざる。そのため、`integrity_review` を独立 stage として入れた。

この stage では主に以下を確認している。

- 募集終了・掲載終了などの stale signal
- 家賃などの数値矛盾
- 売買物件と賃貸物件の取り違え
- 希望エリアとの不一致
- 証拠不足による比較不能候補
- 間取りや駅徒歩の不整合

これは科学論文の reviewer をそのまま再現したものではないが、**ドメイン特化の reviewer layer** と言える。信頼できない候補を ranking 前に落とすことで、最終出力の品質を上げている。

### 5.6 最終出力を Actionable にしたこと
このシステムの出力は単なるランキングではない。最終的に以下までつなげている。

- research summary
- 候補比較表
- 最終 markdown report
- source list
- 物件カード UI
- 問い合わせ文ドラフト
- 契約リスク確認

つまり「探す」で終わらず、「比較する」「問い合わせる」「契約前にリスクを見る」までを一続きの loop にしている。これは practical domain に AI Scientist の考え方を移植するうえで重要な点だと考える。

## 6. AI Scientist / ShinkaEvolve との関係
### AI Scientist v1 との関係
v1 から強く影響を受けたのは、agent が end-to-end に仕事を進めるという考え方である。本プロジェクトでも、条件整理から最終レポートまでを一つの流れとして実装した。

### AI Scientist v2 との関係
v2 からは、tree search によって複数の方向性を探索し、その中から評価に基づいて最善経路を選ぶ発想を取り入れた。今回の branch expansion, pruning, selection は、この影響が最も大きい部分である。

### Automated Reviewer との関係
Nature 論文で述べられている reviewer を、そのまま論文評価器として再現したわけではない。ただし、以下の形で reviewer 的な発想を移植している。

- integrity review による候補信頼性の判定
- offline evaluator による branch quality の採点
- readiness が低い場合に再探索を促す設計

したがって、「論文 reviewer を作った」のではなく、「別ドメインに reviewer の役割を持つ layer を作った」と位置づけるのが正確である。

### ShinkaEvolve との関係
今回の repo では ShinkaEvolve 自体は実装していない。ただし、今は hand-crafted になっている branch operator や ranking profile を、今後進化的に改善する余地がある。ここは将来的に ShinkaEvolve 的な方向へ拡張可能だと考えている。

## 7. システム設計の詳細
### 7.1 Orchestrator の分割
バックエンドは `HousingOrchestrator` を中心に動くが、中身は mixin に分割されている。

- core
- planning
- research
- presentation
- actions
- llm config manager

この分割により、agent loop の責務が比較的明確になっている。単一クラスにすべて詰め込むよりも、Part 5 のような複雑な agent 実装には適した構造だと思う。

### 7.2 Research Journal と Auditability
SQLite には以下の情報が保存される。

- session
- message
- memory
- research job
- research journal node
- audit event
- llm call event
- property catalog

特に `research_journal_nodes` と `audit_events` が重要で、どの branch がどう実行され、なぜ選ばれ、何が落とされたかを後から追える。これは「agent が本当に reasoning しているか」を検証しやすくするために重要だった。

### 7.3 LLM Routing
用途ごとに route を分けている。

- `planner`
- `research_default`
- `communication`
- `risk_check`

また research job 実行中は route 設定を lock している。これにより、途中で model が変わって branch 比較が壊れることを避けている。

### 7.4 LLM Observability
`ObservedLLMAdapter` により、以下が DB に記録される。

- provider
- model
- operation
- prompt / response chars
- token 数
- 推定コスト
- duration
- success / failure

この仕組みを入れた理由は、agent 開発を「勘で prompt をいじる作業」にしないためである。どの route が重いか、どこで失敗しているかを見える化することで、改善可能なシステムにしたかった。

### 7.5 Search 周りの工夫
Brave Search 側には、rate limit 制御と、必要に応じた query rewrite / snippet summarization が入っている。さらに property catalog service 側には mock catalog と semantic scoring があり、完全な open world ではないが、ある程度 controllable な研究環境を用意している。

### 7.6 Frontend の役割
フロントエンドはただの chat 表示ではなく、structured block を描画する。

- plan
- question
- table
- cards
- warning
- timeline
- tree
- sources

branching search の内容をユーザーに説明可能にするため、tree や progress を UI 上で見せる設計にした。これは trust と debugging の両方に効く。

## 8. このプロジェクトの新規性
私はこの実装の新規性を、次の 4 点に整理している。

第一に、AI Scientist の発想を論文生成ではなく、実務的な住宅意思決定ドメインへ移した点。

第二に、物件探索を single-pass retrieval にせず、**branching research problem** として扱った点。

第三に、ranking の前に integrity review を入れ、さらに branch 自体も offline evaluator で選抜することで、二段階の reviewer 的評価を持たせた点。

第四に、探索の結果を、そのまま問い合わせ文や契約リスク確認へ接続し、実行可能な workflow にした点。

## 9. コードベース確認と検証結果
今回、提出ドラフトを書くにあたって、README だけでなく、主要コードとテストを一通り確認した。

確認した主な範囲は以下である。

- `backend/app/main.py`
- `backend/app/orchestrator.py`
- `backend/app/orchestrator_modules/*`
- `backend/app/research/*`
- `backend/app/stages/*`
- `backend/app/services/*`
- `frontend/src/App.tsx`
- `frontend/src/components/StructuredBlock.tsx`
- `backend/tests/*`

### 実装の厚み
`backend/tests` には **92 個の test 関数** があり、planner, orchestrator, research manager, final report, risk check, llm observability など、主要な責務に対してテストが用意されている。PoC ではあるが、かなり意識的に品質管理されている構成だと判断した。

### 実行した検証
この環境で実際に確認できたものは以下である。

- `frontend` での `npm run build`: 成功
- `backend` での `pytest -q`: 収集段階で停止

backend の停止理由は、現在のシェル環境が Python 3.10 である一方、プロジェクト側が Python 3.11 前提で書かれているためである。これは以下の 2 点と一致している。

- `backend/pyproject.toml`: `requires-python = ">=3.11"`
- `backend/Dockerfile`: `FROM python:3.11-slim`

実際のエラーは `from datetime import UTC` で、これは Python 3.11 前提の記法である。したがって、backend テストが壊れていると断定するのではなく、**この端末の実行環境が target runtime と一致していないため、正しい条件での pytest 完走確認は今回のターンではできなかった**と記述するのが正確である。

## 10. 限界
このプロジェクトは PoC であり、明確な限界もある。

- 実データ統合はまだ限定的で、Brave Search と mock catalog に依存している
- branch evaluator は heuristic であり、大規模な人手ラベルで校正された reviewer ではない
- 契約リスク確認は triage として有用だが、法的助言の代替ではない
- 学習や自己改善はまだ限定的で、どの operator が長期的に有効かを自動最適化していない
- 問い合わせ後の返信や成約結果までを loop に取り込めていない

## 11. 今後の発展
今後の発展として、特に重要だと考えるのは以下の 4 点である。

### 11.1 Operator の自動改善
今の branch operator は hand-crafted である。ここを ShinkaEvolve 的な発想で改善し、

- query expansion 戦略
- ranking weight
- branch expansion policy
- prune threshold

を自動的に改善できるようにすると、Part 5 の延長としてかなり面白い研究対象になる。

### 11.2 Housing Reviewer の校正
listing freshness, factual consistency, inquiry usefulness, lease risk severity などに対してラベル付きデータを作り、より校正された reviewer を構築したい。これができると、AI Scientist の reviewer 的発想をさらに強く移植できる。

### 11.3 ドメイン拡張
この設計は賃貸だけでなく、以下にも広げられる。

- 売買物件探索
- 引っ越し比較支援
- 住宅ローン前提の候補比較
- 商業不動産調査
- 日本語文書ベースの他の意思決定支援タスク

### 11.4 Outcome まで含む評価
本当に重要なのは、選ばれた branch が実際に良い結果につながるかである。将来的には、

- stale listing の減少
- shortlist 品質の改善
- 問い合わせ返信率
- 契約リスク回避率

のような outcome 指標まで追い、探索方針を改善できるようにしたい。

## 12. まとめ
Part 5 では、The AI Scientist のアイデアをそのままなぞるのではなく、**計画立案、分岐探索、評価、統合**というコア構造を、日本の賃貸物件調査という別ドメインに移植した。

このシステムが示したいのは、「LLM が部屋探しを会話できる」ということではない。そうではなく、

- 曖昧な要求を research-ready な条件へ変換し
- 複数の探索戦略を並行評価し
- 信頼性の低い候補を reviewer 的に落とし
- 根拠つきで最終推薦をまとめ
- そのまま問い合わせや契約確認へ接続する

という、AI Scientist 型エージェントの再利用可能なパターンである。

その意味で、このプロジェクトは「住宅検索アプリ」ではなく、**AI Scientist 的 agent design を、高摩擦な現実ドメインへ移した実験**として提出したい。
