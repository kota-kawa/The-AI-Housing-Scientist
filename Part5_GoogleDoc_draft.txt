# Part 5: The AI Housing Scientist

AI Scientist の探索・評価構造を、日本の賃貸物件調査エージェントへ移植した概念実証

---

## 1. 概要

本パートでは、Sakana AI の The AI Scientist が持つ中核的な構造――**計画立案・分岐探索・自動評価・統合**――を、機械学習論文の生成ではなく、日本の賃貸物件探索という実務的なドメインへ移植した概念実証（PoC）システム **The AI Housing Scientist** を構築した。

日本の賃貸物件探索を題材に選んだ理由は、この領域が AI Scientist 型のエージェント設計と相性が良いためである。具体的には、情報源が複数のウェブサイトに分散していること、掲載データに欠損や矛盾が含まれやすいこと、家賃・駅距離・間取りなどのトレードオフが複雑であること、そして最終的に問い合わせや契約判断という行動が求められることが挙げられる。これらの性質は、一回の検索で完結させる方式よりも、複数の探索方針を並行して試し、結果を評価して意思決定につなげる方式と適合する。

なお本実装は、AI Scientist の既存コードをそのまま動かしたものではない。論文生成という表面的な用途ではなく、「探索を計画し、分岐を評価し、根拠をまとめて意思決定につなげる」という根本構造を、住宅探索向けに設計し直したものである。

---

## 2. AI Scientist v2 の理解と本実装への応用

AI Scientist v2 では、研究のアイデア生成から実験実行・論文執筆までを自動化するにあたり、木探索（tree search）の仕組みが導入されている。具体的には、複数の研究方針を分岐として生成し、各分岐で実験を実行した後、Automated Reviewer と呼ばれる自動評価器でその成果を採点する。評価の低い分岐は枝刈り（除外）され、有望な分岐がさらに展開される。これにより、「最初に思いついたアイデア」ではなく、「試行の中で最も評価が高かった方向」が最終成果として選ばれる。

この設計の本質は、**生成と評価を分離し、評価が最終的な制御権を持つ**という点にある。本実装ではこの考え方を以下のように賃貸物件探索に応用した。

| AI Scientist v2 の要素 | 本実装での対応 |
|---|---|
| 研究アイデアの分岐生成 | 「条件を厳しくして絞る」「候補数を確保するために緩める」など、異なる探索方針の分岐を生成 |
| 実験の実行 | 各分岐で Web 検索・詳細ページ取得・データ正規化を実行 |
| Automated Reviewer による評価 | 分岐ごとに候補数・詳細取得率・条件一致度などの指標でスコアリングし、品質の低い分岐を除外 |
| 最良分岐の選択と論文生成 | 最も評価の高い分岐の結果をもとに、比較レポート・問い合わせ文・契約リスク確認を生成 |

## 3. システムの全体像と動作例

### 3.1 処理の流れ

システムは以下の流れを一気通貫で処理する。

1. ユーザーが自然文で希望条件を入力する
2. 計画立案モジュールが条件を構造化し、不足があれば追加質問を返す
3. 調査計画を提示し、ユーザーの承認を待つ
4. 承認後、複数の探索方針で分岐型リサーチを実行する
5. 各分岐の候補物件を正規化・重複排除・整合性審査する
6. 分岐を評価・選抜し、最良の結果をランキング・比較レポートにまとめる
7. 選択した候補に対して問い合わせ文の下書きを生成する
8. 契約条項テキストを貼り付けると、リスク（更新料・違約金・解約予告など）を抽出する

### 3.2 具体的な動作例

以下は、実際の利用の流れを示した例である。

**ユーザー入力:**
「新宿駅の近くで、家賃10万円以下の1LDKを探してほしい。ペット可だと嬉しい。」

**Step 1 — 条件の構造化と追加質問:**
システムは入力から「エリア: 新宿駅周辺」「予算: 10万円以下」「間取り: 1LDK」「あると良い条件: ペット可」を抽出し、不足している「入居時期」「駅徒歩の許容範囲」を追加質問として返す。

**Step 2 — 調査計画の提示:**
条件が揃うと、「条件を厳密に絞った検索」「候補数を確保するために条件を少し緩めた検索」「異なる検索表現で情報源を広げた検索」など、複数の探索方針を含む調査計画を提示する。ユーザーが承認すると探索が開始される。

**Step 3 — 分岐探索と評価:**
各探索方針が並行して実行され、それぞれの結果が評価される。たとえば「条件厳密化」の分岐は候補2件・条件一致度が高い、「候補数確保」の分岐は候補5件だが条件一致度はやや低い、といった結果が得られる。評価モジュールがこれらをスコアリングし、最も品質の高い分岐を採用する。

**Step 4 — 最終出力:**
採用された分岐の候補物件が比較表として表示され、各物件の推薦理由が添えられる。ユーザーが候補を選ぶと問い合わせ文の下書きが生成され、契約条項を貼り付ければリスク項目が抽出される。

## 4. 技術的な核心貢献

### 4.1 分岐型の探索設計

最も重要な設計上の選択は、物件探索を一回の検索で完結させず、**分岐型の調査問題**として扱った点である。

分岐探索を管理するリサーチコンポーネントが複数の探索分岐を並行管理し、各分岐に異なる探索方針を適用する。

| 探索方針 | 役割 |
|---|---|
| 条件厳格化 | must 条件と予算一致度を強めて絞り込む |
| 候補拡張 | 候補数と詳細取得率を回復するために条件を緩める |
| 情報源分散 | 異なる検索表現を使い、情報源の偏りを減らす |
| 詳細優先 | 設備・初期費用などの詳細ページの取得を優先する |
| 有望条件の深掘り | スコアの高い分岐の条件組み合わせをさらに深掘りする |
| 近傍条件の探索 | 周辺エリアや「あると良い条件」方面に広げる |

処理の流れは「計画確定 → 分岐探索 → 統合」の 3 段階で構成されている。

### 4.2 分岐の評価と選抜

最も AI Scientist らしい部分は、分岐をただ生成するだけでなく、**分岐を評価して選ぶ仕組み**を入れた点である。各分岐の評価モジュールでは以下の指標を組み合わせて分岐スコア（各分岐の品質を 0〜100 で数値化した指標）を算出し、品質の低い分岐を除外する。

| 評価指標 | 配点 | 内容 |
|---|---|---|
| 候補数 | 20 点 | 正規化後に残った候補の件数 |
| 詳細取得率 | 25 点 | 詳細ページから情報を補完できた候補の割合 |
| 構造化率 | 20 点 | 家賃・駅徒歩・間取りが揃っている候補の割合 |
| 上位候補の条件一致度 | 25 点 | 上位 3 件の条件一致スコアの平均 |
| 情報源の多様性 | 10 点 | 異なる情報源から候補が得られているか |
| 減点: 重複率 | -5 点 | 重複候補が多い場合の減点 |
| 減点: 整合性審査での除外率 | -10 点 | 整合性審査で除外された候補が多い場合の減点 |

最終的に「最後に生成された分岐」ではなく、「評価が最も高い分岐」が採用される。この **生成よりも評価が制御権を持つ** 構造が、AI Scientist の Automated Reviewer の考え方と直接対応している。

### 4.3 整合性審査によるドメイン特化の品質管理

賃貸領域では、古い掲載・価格矛盾・希望エリア外の物件などが検索結果に混入しやすい。そのため整合性審査のステージをランキングの前に独立して設け、以下の観点で候補を検証している。

- **掲載鮮度:** 「募集終了」「成約済」などの表記を検出し、古い物件を除外
- **価格整合性:** タイトルと詳細ページで家賃が矛盾していないかを確認
- **物件種別:** 売買物件が賃貸検索に混入していないかを判定
- **エリア一致:** 希望エリアと実際の所在地が一致しているかを確認

この審査はルールベースの判定と LLM による補助判定を組み合わせて行い、各候補に信頼度スコア（0〜100）を付与する。これは「論文の Reviewer を作った」のではなく、「別ドメインに Reviewer の役割を持つ品質管理の層を作った」と位置づけるのが正確である。

### 4.4 出力を行動につなげる設計

最終出力は候補のランキングだけでなく、調査要約・候補比較表・推薦理由に加え、問い合わせ文の下書きと契約リスクの確認まで含む。「探す」で終わらず、「比較する」「問い合わせる」「契約前にリスクを確認する」まで一つの処理の流れにしている。実務的なドメインに AI Scientist の考え方を移植するうえで、出力が行動に直結することは重要だと考える。

### 5.1 対応モデル

本システムは複数の LLM プロバイダに対応しており、用途別にモデルを切り替えるルーティング機構を持つ（計画立案・リサーチ・コミュニケーション・リスク確認の 4 系統）。対応するプロバイダとデフォルトモデルは以下の通りである。

### 5.2 技術構成

| 要素 | 内容 |

LLM 呼び出しの監視レイヤーにより、モデル名・トークン数・推定コスト・処理時間が DB に自動記録され、どの処理でコストがかかっているかを定量的に把握できる。

## 6. 先行事例との差別化

既存の不動産 AI サービス（例: 物件提案チャットボットや条件検索の自然言語インターフェース）は、基本的に一回の検索結果を整形して返す構造であり、複数の探索方針を並行して評価する仕組みは持っていない。また、AI Scientist のコードをそのまま不動産に適用した事例は確認できなかった。

本実装の差別化は、「賃貸 AI チャットボット」を作ったことではなく、AI Scientist の**分岐探索・自動評価・枝刈りという構造**を不動産という別ドメインに移植し、さらに整合性審査や問い合わせ文生成・契約リスク確認までを一つの処理ループに組み込んだ点にある。

---

## 7. 限界と今後の展望

**現状の限界:**

- 実データ統合はまだ限定的で、Web 検索と疑似物件データベースに依存している
- 分岐の評価は経験則ベースの指標であり、大規模なラベル付きデータで校正された評価器ではない
- 契約リスク確認は一次的な確認として有用だが、法的助言の代替ではない
- 各探索方針は手動で定義されており、自動的に改善される仕組みはまだない

**最も重要な次ステップ:**

① 探索方針の自動改善: 検索クエリの拡張戦略やランキングの重み付けを、ShinkaEvolve の発想で進化的に最適化する。ShinkaEvolve は LLM を用いてプログラムを進化的に改善するフレームワークであり、現在手動で定義している探索方針の改善に適用できると考えている。

② ドメイン特化の評価器の構築: 物件掲載の鮮度・事実整合性・問い合わせ有用性などに対してラベル付きデータを作成し、より精度の高い品質評価器を構築する。これにより、AI Scientist の Automated Reviewer の発想をより強く反映できる。

---

## 8. まとめ

本実装は「住宅検索アプリ」として提出するのではなく、AI Scientist 型のエージェント設計――曖昧な要求の構造化・分岐探索・自動評価による選抜・行動につながる出力の接続――を、情報が不完全で意思決定が必要な現実のドメインへ移植した実験として位置づけている。

このプロジェクトが示したいのは、AI Scientist の発想が ML 論文生成に限定されず、探索と評価が必要な他のドメインにも再利用可能なパターンである、という点である。

### A-1. 分岐の評価スコアリング

各探索分岐の品質を数値化する関数の中核部分である。候補数・詳細取得率・構造化率・上位候補の条件一致度・情報源の多様性の 5 指標を加点し、重複率と整合性審査での除外率を減点する。

```python
def evaluate_branch(
    *,
    branch_id: str,
    label: str,
    raw_results: list[dict[str, Any]],
    normalized_properties: list[dict[str, Any]],
    ranked_properties: list[dict[str, Any]],
    duplicate_groups: list[dict[str, Any]],
    search_summary: dict[str, Any],
    parent_summary: dict[str, Any] | None = None,
    # ... 他の引数は省略
) -> dict[str, Any]:
    normalized_count = len(normalized_properties)
    detail_coverage = round(detail_hit_count / detail_denominator, 3)
    structured_ratio = _structured_property_ratio(normalized_properties)

    # 上位3件の条件一致スコアの平均を算出
    if ranked_properties:
        top_scores = [float(item.get("score") or 0.0) for item in ranked_properties[:3]]
        avg_top3_score = round(sum(top_scores) / len(top_scores), 3)

    # 情報源の多様性（異なる情報源の数）
    source_diversity = len({
        str(item.get("source_name") or "")
        for item in raw_results
        if str(item.get("source_name") or "").strip()
    })

    # 5指標の加点と2指標の減点でスコアを算出
    score = 0.0
    score += _score_from_range(normalized_count, 0, 6) * 20.0       # 候補数
    score += detail_coverage * 25.0                                  # 詳細取得率
    score += structured_ratio * 20.0                                 # 構造化率
    score += _score_from_range(avg_top3_score, 35.0, 95.0) * 25.0   # 上位候補一致度
    score += _score_from_range(source_diversity, 1, 3) * 10.0       # 情報源多様性
    score -= duplicate_ratio * 5.0                                   # 重複率による減点
    score -= integrity_drop_ratio * 10.0                             # 整合性除外による減点

    # 評価結果に基づく課題と改善提案を生成
    issues: list[str] = []
    recommendations: list[str] = []
    if detail_coverage < 0.4:
        issues.append("詳細ページ補完率が低い")
        recommendations.append("詳細ページ取得を優先する検索分岐を残す")
    if source_diversity <= 1:
        issues.append("情報源の多様性が低い")
        recommendations.append("Brave と catalog の両系統で候補を確保する")
    # ...
```

### A-2. 整合性審査（ランキング前の品質フィルター）

各候補物件に対してルールベースの検証と LLM による補助検証を行い、信頼度の低い候補をランキング前に除外する関数である。

```python
def run_integrity_review(
    *,
    normalized_properties: list[dict[str, Any]],
    adapter: LLMAdapter | None = None,
    target_area: str = "",
    listing_type: str = "",
    # ... 他の引数は省略
) -> dict[str, Any]:
    # Step 1: 各候補にルールベースの検証を実施
    for prop in normalized_properties:
        rule_review = _rule_review_for_property(
            prop=prop,
            today=resolved_today,
            target_area=target_area,
            listing_type=listing_type,
        )

    # Step 2: LLM による補助的な検証（利用可能な場合）
    if adapter is not None:
        llm_reviews_by_id = _build_llm_integrity_reviews(
            normalized_properties=normalized_properties,
            adapter=adapter,
        )

    # Step 3: ルール判定とLLM判定を統合し、除外判定を行う
    for prop in normalized_properties:
        merged_review = _merge_reviews(
            rule_reviews_by_id.get(property_id, {}),
            llm_reviews_by_id.get(property_id),
        )
        # 信頼度スコア(0〜100)が低い候補、またはshould_drop=Trueの候補を除外
        if merged_review["should_drop"]:
            dropped_properties.append(prop)
        else:
            filtered_properties.append(prop)
```

### A-3. 探索方針ごとの検索クエリ生成

各探索方針に応じて、ユーザーの希望条件から異なる切り口の検索クエリを生成するロジックである。

```python
# 条件厳格化: must条件・予算・駅徒歩を明示して絞り込む
if operator == "tighten_match":
```
